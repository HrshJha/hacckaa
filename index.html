04 — AI Approach

Teaching Claude to
speak Hinglish.

The hardest problem isn't building the pipeline. It's making an LLM reliably distinguish "bhaiya" from a customer name, an inquiry from a confirmed order, and "kal tak" from a negotiation. Here's how we solved it.

01

Input Normalization

Voice note transcripts, typed Hinglish, and abbreviations are preprocessed to normalize encoding, fix common OCR errors from screenshots, and standardize mixed-script inputs before prompting.

02

Intent Classification

First pass distinguishes inquiry ("price kya hai?") from confirmed order ("book kar do"), negotiation from acceptance, and casual conversation from actionable business content. Prevents false positives.

03

Structured Extraction with Schema

Claude is constrained to a strict JSON schema: items[], quantities[], units, delivery_date, customer_identity, special_instructions. Deterministic temperature (0) ensures consistent, parseable output every time.

04

Few-Shot Hinglish Examples

The system prompt includes 8 curated examples of messy Indian SMB chat — covering textile, grocery, D2C, and service businesses — with correct extractions. Accuracy jumped from 78% to 94% after this change alone.

05

Human Verification Gate

Before invoice generation, a human-in-the-loop verification layer surfaces low-confidence extractions. Business owner confirms in one tap. Ensures trust before any financial document is generated.

06

Revenue Anticipation (Phase 2)

Pattern learning across order history builds a per-customer purchase graph. FlowChat surfaces predictions — seasonal buying patterns, reorder cycles, at-risk customers — proactively, not reactively.

08 — Privacy Considerations

Conversations are
sacred data.

FlowChat processes the most sensitive business data imaginable — customer identities, purchase patterns, financial flows. Privacy is architecture, not an afterthought.

🔒

PII Auto-Redaction

Middleware automatically masks phone numbers and customer names in all logs, error reports, and API responses. No personally identifiable information appears in observability tooling or Sentry traces.

🏠

Data Residency

All business data — orders, invoices, customer records — is stored in the business owner's own PostgreSQL instance. FlowChat does not aggregate or share data across businesses. Each deployment is isolated.

🔑

Minimal Data Principle

FlowChat extracts only what's necessary for order management — items, quantities, delivery intent. It does not store full conversation transcripts, voice note audio, or metadata beyond the extracted order object.

🛡️

Secure Transmission

All API communication is encrypted in transit. Helmet headers enforce strict content security policies. Input sanitization prevents injection attacks. Rate limiting blocks enumeration attempts.

👁️

Consent Architecture

The human verification gate means every extracted order is reviewed by the business owner before any action is taken. No invoice is generated or message sent without explicit human approval in Phase 1.

📋

DPDP Compliance Path

India's Digital Personal Data Protection Act 2023 alignment is built into the architecture: purpose limitation, data minimization, and consent-based processing are core to FlowChat's design — not retrofitted.
